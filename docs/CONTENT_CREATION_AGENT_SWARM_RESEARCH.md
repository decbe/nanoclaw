### 2.2 角色详细定义

#### Lead Agent (主编/CMO)
```markdown
职责：
- 每日热点选题策划
- 任务分配和进度跟踪
- 内容质量最终审核
- 策略调整和优化

工具：
- WebSearch：热点搜索
- TaskCreate/TaskUpdate：任务管理
- SendMessage：团队协调

输出：
- 每日选题清单
- 任务分配指令
- 质量审核反馈
```

#### Researcher (研究员)
```markdown
职责：
- 搜索24小时内热点新闻
- 收集相关资料和数据
- 竞品内容分析
- SEO 关键词研究

工具：
- WebSearch：新闻搜索
- WebFetch：内容提取
- arxiv-mcp：论文检索（可选）

输出：
- 选题研究报告
- 素材清单（含来源链接）
- 关键词建议
```

#### Writer (作者)
```markdown
职责：
- 根据选题撰写初稿
- 保持品牌写作风格
- 遵守反AI检测规则
- Markdown 格式输出

风格要求：
- 信息密度高
- 禁用过渡词（首先、其次、总之）
- 避免评价性语言
- 数据支撑观点

输出：
- 完整文章草稿（Markdown）
- 配图建议
```

#### Editor (编辑)
```markdown
职责：
- 文字校对和润色
- 图片筛选和处理
- 排版美化
- 发布到各平台

工具：
- wenyan-mcp：公众号发布
- Read：图片验证
- Bash/curl：图片下载

输出：
- 最终定稿
- 发布确认
```

#### Distributor (分发员) - 可选
```markdown
职责：
- 同步分发到多平台
- 平台特定格式适配
- 发布时间调度
- 数据统计反馈

支持平台：
- 微信公众号（主）
- 知乎专栏
- 头条号
- 百家号
- 掘金
- CSDN
```

## 三、工作流设计

### 3.1 线性流水线模式

```
选题 → 研究 → 撰写 → 审核 → 发布
  │      │      │      │      │
  └──────┴──────┴──────┴──────┘
         顺序执行，一个完成传下一个
```

**优点**：简单清晰，易于调试
**缺点**：效率较低，无法并行

### 3.2 迭代优化模式

```
     ┌──────────────────────────────┐
     │                              │
     ▼                              │
  撰写 → 审核 → [不通过] ───────────┘
     │
     ▼ [通过]
   发布
```

**优点**：质量可控
**缺点**：迭代次数不确定

### 3.3 时间槽调度模式（推荐）

```
┌─────────────────────────────────────────────────────────────┐
│                     Weekly Plan                              │
├─────────┬─────────┬─────────┬─────────┬─────────┬──────────┤
│ 时间槽   │ 周一    │ 周二    │ 周三    │ 周四    │ 周五     │
├─────────┼─────────┼─────────┼─────────┼─────────┼──────────┤
│ 09-12   │ 选题+研究│ 撰写    │ 选题+研究│ 撰写    │ 审核+发布 │
├─────────┼─────────┼─────────┼─────────┼─────────┼──────────┤
│ 14-17   │ 撰写    │ 审核    │ 撰写    │ 发布    │ 数据分析  │
├─────────┼─────────┼─────────┼─────────┼─────────┼──────────┤
│ 19-22   │ 社媒互动 │ 社媒互动 │ 社媒互动 │ 社媒互动 │ 周报     │
└─────────┴─────────┴─────────┴─────────┴─────────┴──────────┘
```

**优点**：可预测、可并行、资源利用高效
**缺点**：需要提前规划

## 四、微信公众号发布方案

### 4.1 方案对比

| 方案 | 实现方式 | 优点 | 缺点 |
|------|---------|------|------|
| **wenyan-mcp** | 官方 API | 稳定可靠 | 需要认证公众号 |
| **ChatWiki** | RAG + API | 开箱即用 | 功能有限 |
| **扣子 Coze** | 工作流 | 低代码 | 依赖平台 |
| **腾讯元器** | 官方平台 | 深度集成 | 闭源 |

### 4.2 wenyan-mcp 配置

```json
{
  "wenyan-mcp": {
    "command": "node",
    "args": ["/path/to/wenyan-mcp/dist/index.js"],
    "env": {
      "WECHAT_APP_ID": "YOUR_APP_ID",
      "WECHAT_APP_SECRET": "YOUR_APP_SECRET"
    }
  }
}
```

获取凭证：微信公众平台 → 开发 → 基本配置

### 4.3 发布流程

```
1. Writer 完成 Markdown 文章
2. Editor 调用 wenyan-mcp 创建草稿
3. 草稿进入公众号后台
4. 人工最终审核
5. 手动/定时发布
```

## 六、NanoClaw 实现方案

### 6.1 基于现有架构

NanoClaw 已具备 Agent Swarm 能力，**无需修改源码**，只需配置：

```
1. 运行 /add-telegram-swarm 获取 Bot Pool（可选）
2. 配置 groups/{群组}/CLAUDE.md 定义团队
3. 配置 wenyan-mcp 发布到公众号
```

### 6.2 CLAUDE.md 配置示例

```markdown
# Content Creation Team

你是一个内容创作团队的主编。根据任务需要创建队友：

## 团队成员

1. **Researcher (研究员)**
   - 搜索热点、收集资料、分析竞品
   - 输出：选题报告 + 素材清单

2. **Writer (作者)**
   - 撰写文章、保持风格、反AI检测
   - 输出：Markdown 文章

3. **Editor (编辑)**
   - 校对润色、图片筛选、排版发布
   - 输出：最终定稿 + 发布

## 协作规则

每个队友必须：
- 使用 `mcp__nanoclaw__send_message` 向群组汇报进度，sender 参数设为自己的角色名
- 保持消息简短（2-4句话）
- 使用 `SendMessage` 与队友内部协调

## 写作风格

- 信息密度高，避免空洞表达
- 禁用过渡词：首先、其次、最后、总之
- 数据支撑观点，引用来源
- 段落简短，每段不超过150字

## 发布流程

1. Editor 收到最终稿件
2. 调用 wenyan-mcp 创建公众号草稿
3. 返回草稿链接供人工审核
```

### 6.3 调度方式

```bash
# 手动触发
claude -p "执行今日内容创作任务"

# 定时调度（cron）
0 9,14,19 * * * cd /home/bbc/apps/nanoclaw && claude -p "执行当前时间槽的内容创作任务"
```
